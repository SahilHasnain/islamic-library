# For You Sorting (Curated Recommendations)

This document defines a curated "For You" sorting algorithm for the Library screen.

## Problem

The Library list is currently sorted by simple heuristics (e.g., alpha, recently read). Users often want a guided "what should I read next" experience that does not require them to decide from scratch.

We also want admin control over recommendations so the reading journey reflects intentional curation rather than a brittle ML model.

## High-Level Solution

Add a curated recommendation field per book in the admin console. Publish it into the public `catalog.json`. On the client, compute a default sort order called **For You**.

This keeps discovery smarter while preserving the value of the Book Detail page (users still land on details when selecting a book).

## Goals

- Make the default Library sort feel personalized and guided.
- Use admin curation as the primary recommendation signal.
- Always prioritize continuing in-progress reading.
- Avoid complex ML or on-device training.
- Keep the algorithm explainable and debuggable.

## Non-Goals

- Fully automatic recommendations from embeddings/ML.
- Replacing existing filters (category, language, search).
- Designing the full admin ingestion/publishing pipeline (beyond data fields and expectations).

## Data Model

### V1 (Minimal)

Add a single field on each book:

- `nextRecommendedBookId?: string`

Constraints:

- Must be empty or a valid book ID from the catalog.
- Must not equal its own `id`.
- (Nice-to-have) warn if it creates obvious cycles (A -> B -> A).

### Public Catalog Types

Extend `PublicCatalogBook`:

```ts
export type PublicCatalogBook = {
  id: string;
  title: string;
  subtitle?: string;
  author?: string;
  category?: string;
  categoryLabel?: string;
  tags?: string[];
  coverImage?: string;
  status: "published";
  metadataUrl: string;
  nextRecommendedBookId?: string; // NEW
};
```

Rationale: recommendations should not require fetching per-book metadata files.

## Admin Console UX

For each book record:

- Add a dropdown: **Next recommended book**
- Options: all published books (title + id)
- Value stored as `nextRecommendedBookId`

Validation:

- Disallow selecting the current book.
- Optional warning if selecting creates a cycle.

## Client Algorithm

### Terminology

- **In-progress** book: has `latestProgressByBook[bookId]` and is not completed.
- **Anchor book**: the user's most relevant recent book; used to seed the recommendation chain.
- **Next Up chain**: sequence generated by following `nextRecommendedBookId`.

### Inputs

- `catalog.books` (includes `nextRecommendedBookId`)
- `latestProgressByBook` (recency + current book)
- `completedBookIds`
- (Optional) `activePlanMap` for small boosts

### Default Sort: For You

Order the visible list in three phases:

1. **Continue Reading**
   - Books in progress
   - Sorted by most recently read (`updatedAt` desc)

2. **Next Up** (curated)
   - Determine the **anchor book**:
     - Prefer the most recently read book (top of Continue Reading)
     - If none, fallback to the most recently completed book (if tracked)
     - If none, no anchor
   - Follow the anchor's `nextRecommendedBookId` to produce a queue:
     - `anchor -> next -> next -> ...`
     - Stop when:
       - missing/invalid id
       - repeats (cycle detection)
       - reaches a max length (e.g., 10)
   - Filter out books already shown in phase 1.
   - (Recommended) demote completed books (either filter them out or move them later).

3. **Everything Else**
   - Remaining books not already shown.
   - Stable fallback ordering:
     - optional: featured first (if available)
     - optional: same category/tags as anchor (if anchor exists)
     - final fallback: alphabetical by title

### Properties

- Deterministic: same inputs yield same order.
- Explainable: we can show "Because you read X".
- Cheap: O(n) for indexing + O(k) chain traversal.

## V2 (Optional Upgrade)

Evolve from a single next-book link to a small ranked list:

```ts
type BookRecommendation = {
  bookId: string;
  weight: number; // e.g. 0..100
  reasonLabel?: string;
};

type PublicCatalogBook = {
  // ...
  recommendations?: BookRecommendation[];
};
```

Scoring approach:

- Strong boost if directly recommended by anchor.
- Medium boost if shares category/tags with anchor.
- Demote completed.
- Small boost if the edition/language is available.

Optional lightweight personalization (no ML):

- If a user repeatedly ignores a recommended book (never opens it after being surfaced), reduce its local weight.

## Edge Cases

- Invalid `nextRecommendedBookId`: ignore.
- Cycles: detect via visited set and stop.
- Anchor is completed: still OK; can use it for Next Up but demote already-completed targets.
- Filters/search: apply filters first, then apply For You ordering to the filtered set.

## Rollout Plan

1. Add `nextRecommendedBookId` to admin console and publishing pipeline into `catalog.json`.
2. Extend app `PublicCatalogBook` type.
3. Implement "For You" sort and make it the default sort option in Library.
4. Add instrumentation (optional): track open rate of recommended books.

## Success Criteria

- Increased book opens from the Library list.
- Higher completion rate of recommended sequences.
- Lower time-to-first-open for new users.
